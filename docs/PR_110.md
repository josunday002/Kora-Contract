### Implementation Details

#### Status Transition State Machine
The contract enforces this exact transition path:
```
Created
  └─→ Listed (by marketplace)
       └─→ Funded (by financing_pool)
            ├─→ Repaid (by financing_pool, on full repayment)
            └─→ Defaulted (by admin, after due_date)
```

Tests verify:
- No forward skipping (Created → Funded fails)
- No backward transitions (cannot go from Funded → Listed)
- No lateral transitions (cannot go from Listed → Defaulted)

#### Immutability Enforcement
The following fields are immutable after creation:
- `id` - Invoice ID (never changes)
- `sme` - Invoice creator address
- `debtor_hash` - SHA-256 hash of debtor info
- `amount` - Invoice amount
- `currency` - Token symbol
- `due_date` - Payment deadline
- `ipfs_cid` - IPFS metadata reference
- `created_at` - Creation timestamp
- `risk_score` - Initial risk score
- `risk_tier` - Calculated risk tier

Mutable fields (status, funded_at, repaid_at) are changed through explicit state transitions only.

#### Migration Strategy
Migration function is:
- **Idempotent**: Safe to call multiple times
- **Preserving**: Existing invoices remain unchanged
- **Versioned**: Supports future schema upgrades

Version path: 0 → 1 (current), with hooks for future versions.
